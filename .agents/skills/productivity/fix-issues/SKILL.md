# Fix-Issues

이 스킬은 프로젝트의 코드 품질 검증 워크플로우(`/verify-and-fix` 등) 실행 중 발견된 에러(Lint, Type, Test, Build)를 일관성 있고 효과적으로 해결하기 위한 가이드라인입니다.

## 처리 원칙

에러를 수정할 때는 다음 원칙을 준수합니다.

1. **원인 분석 우선**: 에러 메시지와 스택 트레이스를 주의 깊게 읽고 근본적인 원인을 먼저 파악합니다.
2. **최소한의 변경**: 발생한 에러를 수정하는 데 필요한 최소한의 코드 변경만 수행하여 사이드 이펙트를 방지합니다.
3. **규칙 준수**: Lint 및 Type 오류 수정 시 프로젝트의 설정 및 컨벤션을 강제로 무시(`@ts-ignore`, `eslint-disable`)하지 않고 올바른 타입과 포맷으로 수정합니다.

## 에러 종류별 해결 가이드

### 1. Formatting & Linting Errors (`npm run lint`)

- **자동 픽스 시도**: 항상 `npm run lint:fix`를 먼저 실행하여 자동으로 수정 가능한 부분(예: 따옴표, 들여쓰기, 세미콜론 등)을 해결합니다.
- **수동 픽스**: 자동 해결이 안 되는 에러는 규칙 이름을 확인하고 올바른 코드 패턴으로 리팩토링합니다.
- **의존성 배열**: `useEffect`, `useCallback` 등의 React Hook 의존성 배열 경고는 의도한 로직에 맞춰 꼼꼼히 추가하거나 구조를 재검토합니다.

### 2. Type Checking Errors (`npm run type-check`)

- **타입 추론 활용**: 명시적 타입 선언보다 TypeScript의 타입 추론을 최대한 활용합니다.
- **`any` 지양**: `any` 타입 사용을 피하고, 정확한 인터페이스나 타입을 정의하거나 제네릭(Generics)을 활용합니다.
- **Nullish 처리**: `undefined`나 `null`이 될 수 있는 값에 대한 접근은 옵셔널 체이닝(`?.`)이나 조건부를 통해 안전하게 접근합니다.

### 3. Unit Test Failures (`npm run test`)

- **테스트 코드와 소스 코드 대조**: 실패한 테스트의 기대 결과(Expected)와 실제 결과(Received)를 확인하고, 테스트 케이스가 잘못된 것인지 제품 코드가 잘못된 것인지 판단합니다.
- **격리된 테스트**: 다른 테스트의 상태에 영향을 받아 실패하는 Flaky Test일 경우, `beforeEach`, `afterEach` 등을 이용해 상태나 Mock을 초기화합니다.
- **Mocking 재검토**: 외부 의존성(예: Drizzle ORM, Redis 등)이 올바르게 Mocking 되었는지, 반환값이 실제 환경과 유사한지 확인합니다.

### 4. Build Errors (`npm run build`)

- **의존성 충돌**: 모듈을 찾을 수 없거나 버전 충돌이 일어나는 경우, `package.json` 및 `node_modules` 설치 상태를 점검합니다.
- **번들러/컴파일러 로그**: Next.js (또는 사용 중인 프레임워크)의 빌드 에러 로그를 꼼꼼히 읽어, 사용 불가능한 API 호출(예: 서버 컴포넌트 내에서의 클라이언트 훅 사용 등)이 있는지 확인합니다.

## 워크플로우와의 연계

이 스킬은 직접 실행되기보다는, 다른 에이전트 작업이나 `/verify-and-fix` 워크플로우 수행 시 에러가 발생했을 때 참조되는 **지침(Guideline)** 역할을 합니다. 에러 발생 시 이 문서의 원칙과 가이드에 따라 디버깅과 수정을 진행하십시오.
